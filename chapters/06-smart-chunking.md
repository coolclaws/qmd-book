# 第 6 章：智能分块算法

> 将一篇长文档切成大小合适、语义完整的片段，听起来简单，做起来却充满取舍——切太小丢上下文，切太大超 token 限额，切在错误的位置则会把一个完整的段落拦腰截断。qmd 的智能分块系统用一套"带衰减的加权断点评分"算法，优雅地解决了这个问题。

## 常量体系：双轨制的尺寸控制

打开 `src/store.ts`，最先映入眼帘的是一组精心调校的常量（L50–59）：

```ts
// 文件: src/store.ts L50-59
CHUNK_SIZE_TOKENS = 900
CHUNK_OVERLAP_TOKENS = 135   // 15%
CHUNK_SIZE_CHARS = 3600
CHUNK_OVERLAP_CHARS = 540
CHUNK_WINDOW_TOKENS = 200
CHUNK_WINDOW_CHARS = 800
```

这里存在两套平行的度量单位——token 和字符。为什么需要两套？因为 token 计数依赖分词器，调用一次分词器的开销不算小。qmd 的策略是：先用字符做快速估算（按 1 token ≈ 4 chars 的保守比例），只有在需要精确控制时才真正调用分词器。

重叠比例固定在 15%。这个数字在信息检索领域是一个经验值——太小会导致跨块查询命中率下降，太大则浪费存储和计算。`CHUNK_WINDOW` 定义了在目标切割点前后的搜索窗口，分块器会在这个窗口内寻找最佳断点，而不是粗暴地在固定位置一刀切。

## 核心数据结构

### BreakPoint 接口

```ts
// 文件: src/store.ts L76-80
interface BreakPoint {
  pos: number
  score: number
  type: string
}
```

每个 `BreakPoint` 记录三件事：文档中的字符位置、该断点的质量评分、以及断点类型（如 `h2`、`blank` 等）。评分越高，表示在这个位置切割越"干净"——读者不会感到突兀。

### CodeFenceRegion 接口

```ts
// 文件: src/store.ts L86-89
interface CodeFenceRegion {
  start: number
  end: number
}
```

代码围栏区域用于标记 ``` 包裹的代码块。分块器绝不会在代码块内部切割，因为那会破坏代码的完整性，导致语法高亮失败，甚至改变语义。

## 断点模式与优先级

`BREAK_PATTERNS` 数组（L97–110）是整个分块系统的"知识库"，它定义了 Markdown 文档中所有可能的自然断点及其优先级：

| 模式 | 匹配内容 | 评分 |
|------|---------|------|
| h1 | `# 标题` | 100 |
| h2 | `## 标题` | 90 |
| h3 | `### 标题` | 80 |
| codeblock | ` ``` ` | 80 |
| h4 | `#### 标题` | 70 |
| hr | `---` 或 `***` | 60 |
| h5 | `##### 标题` | 60 |
| h6 | `###### 标题` | 50 |
| blank | 空行 | 20 |
| list | `- ` 或 `* ` | 5 |
| numlist | `1. ` | 5 |
| newline | 换行符 | 1 |

这个优先级设计反映了 Markdown 文档的语义层次。一级标题是最强的分界线，分数 100；代码块边界和三级标题并列为 80，因为代码块的开始/结束同样是强语义边界。空行得 20 分，是"可以切但不是最佳选择"的位置。列表项仅 5 分，因为列表内部通常是紧密关联的内容。换行符 1 分，是最后的退路——总比切在一个单词中间好。

值得注意的是，代码块（codeblock）的评分和 h3 一样高。这是因为代码块的起止标记本身就是天然的段落分界，在这里切割既不会破坏代码，又能保持块的语义独立性。

## 扫描与检测

### scanBreakPoints 函数

`scanBreakPoints()`（L117–138）负责对整篇文档执行一次全量扫描，找出所有潜在的断点。它遍历 `BREAK_PATTERNS` 中的每个正则表达式，对文档执行全局匹配。当同一个位置被多个模式命中时（例如一个空行恰好在标题前面），只保留评分最高的那个。最终结果按位置排序返回。

这个"一次扫描、全局复用"的设计很重要。在后续的分块循环中，每次寻找切割点时不需要重新扫描文档，只需在预计算的断点数组中做区间查询即可。

### findCodeFences 函数

`findCodeFences()`（L144–166）追踪文档中所有代码围栏的开闭状态。它从头到尾扫描 ``` 标记，遇到开标记记录起点，遇到闭标记记录终点。如果文档末尾有未闭合的围栏，该区域会自动延伸到文档结尾——这是一种防御性设计，避免因格式错误导致分块器在代码内部切割。

### isInsideCodeFence 函数

`isInsideCodeFence()`（L171–173）是一个简洁的辅助函数，判断给定位置是否落在任何代码围栏区域内。如果是，该位置就不能作为切割点，无论它的断点评分有多高。

## 核心算法：findBestCutoff

`findBestCutoff()`（L188–224）是整个分块系统最精妙的部分。它的任务是：在目标切割位置附近的窗口内，找到一个"综合最优"的断点。

### 平方距离衰减

算法的核心思想是：断点离目标位置越远，它的有效评分越低。但这个衰减不是线性的，而是平方的：

```
normalizedDist = distance / windowSize    // 归一化到 [0, 1]
multiplier = 1.0 - (normalizedDist ^ 2) * decayFactor
```

其中 `decayFactor = 0.7`。这意味着在窗口内不同位置的乘数如下：

| 距目标的位置 | 归一化距离 | 乘数 |
|------------|----------|------|
| 目标位置 | 0% | 1.000 |
| 窗口 25% 处 | 0.25 | 0.956 |
| 窗口 50% 处 | 0.50 | 0.825 |
| 窗口 75% 处 | 0.75 | 0.606 |
| 窗口边缘 | 1.00 | 0.300 |

最终评分 = 断点基础分 × 乘数。

### 为什么用平方而不是线性？

线性衰减（`1 - dist * decay`）的问题是衰减速度恒定。一个 h2 标题（90 分）在窗口 50% 处会被线性衰减削弱到 `90 × 0.65 = 58.5`，这可能低于目标位置附近的空行（`20 × 1.0 = 20`）。但事实上，在 h2 标题处切割几乎总是更好的选择。

平方衰减则更温和——在窗口前半段衰减很慢，后半段才急剧下降。同样的 h2 在 50% 处得分是 `90 × 0.825 = 74.25`，远高于任何低质量断点。这确保了高质量的语义断点即使距离较远，仍然能击败目标位置附近的低质量断点。

代码中还会跳过落在代码围栏内的断点，确保不会在代码块中间切割。

## 字符级分块器：chunkDocument

`chunkDocument()`（L2024–2085）是面向字符的主分块函数。它的工作流程如下：

首先，预扫描整篇文档，生成断点列表和代码围栏区域列表。这两个数据结构在整个分块过程中复用，避免重复计算。

然后进入主循环：从当前位置出发，向前推进 `CHUNK_SIZE_CHARS` 个字符作为目标切割点，调用 `findBestCutoff()` 在窗口内寻找最佳断点。找到后，从当前位置到断点位置的文本成为一个块。下一个块的起点则回退 `CHUNK_OVERLAP_CHARS` 个字符，形成重叠区域。

重叠区域的存在保证了跨块边界的内容不会"消失"。如果用户搜索的关键词恰好跨越两个块的边界，至少有一个块能完整包含它。

## Token 级分块器：chunkDocumentByTokens

`chunkDocumentByTokens()`（L2091–2137）在字符级分块器的基础上增加了 token 精度控制。它采用两阶段策略：

第一阶段：用保守的字符估算（3 chars/token，比通常的 4 更保守）调用字符级分块器，得到初步的分块结果。

第二阶段：对每个块调用真正的 LLM 分词器计算 token 数。如果某个块仍然超过 `CHUNK_SIZE_TOKENS`，则对该块单独再次分割。

这种两阶段方法在精度和性能之间取得了平衡。大多数块在第一阶段就满足 token 限制，只有少数边界情况需要第二阶段的修正。保守的 3 chars/token 估算确保第一阶段宁可切得小一点，也不要产生太多超限的块。

## 本章小结

qmd 的智能分块系统是一个精心设计的多层架构。常量体系提供了字符和 token 的双轨控制；`BREAK_PATTERNS` 编码了 Markdown 的语义层次知识；`scanBreakPoints()` 和 `findCodeFences()` 通过一次性预扫描避免重复计算；`findBestCutoff()` 用平方距离衰减在"切割质量"和"位置偏好"之间找到最优平衡；两级分块器则在性能和精度之间取得了务实的折中。整套设计的核心洞察是：分块不是一个纯数学问题，而是一个需要理解文档语义结构的信息检索问题。
