# 第 7 章：BM25 全文搜索

> 向量搜索擅长捕捉语义相似性，但当用户输入精确的函数名、错误代码或文件路径时，传统的全文搜索依然是最可靠的武器。qmd 基于 SQLite FTS5 构建了一套完整的 BM25 搜索管线，从自定义查询语法到分数归一化，每一层都有值得细读的设计。

## FTS5 虚拟表：搜索的基石

qmd 在 SQLite 中创建了一张 FTS5 虚拟表来支撑全文索引（`src/store.ts` L738–742）：

```sql
-- 文件: src/store.ts L738-742
CREATE VIRTUAL TABLE IF NOT EXISTS documents_fts USING fts5(
    filepath, title, body,
    tokenize='porter unicode61'
)
```

这张表索引了三个字段：`filepath`（文件路径）、`title`（标题）和 `body`（正文）。分词器的选择是 `porter unicode61`——这是两个分词器的组合。`unicode61` 负责按 Unicode 6.1 标准分割文本，正确处理非 ASCII 字符；`porter` 则在其上层应用 Porter 词干提取算法，将 "running"、"runs"、"ran" 统一还原为 "run"。

### 为什么选择 Porter 词干提取？

Porter 算法是信息检索领域最经典的英文词干提取器。它通过一系列规则剥离后缀，使得同一词根的不同形态能够匹配。对于代码文档搜索来说，这意味着搜索 "configure" 也能命中包含 "configuration" 或 "configured" 的文档。代价是偶尔会产生过度归并（如 "university" 和 "universal" 可能被归为同一词干），但在技术文档场景下这种误差可以接受。

`unicode61` 的加入确保了中文、日文等多字节字符不会被错误切分。虽然 Porter 算法本身只对英文有效，但 `unicode61` 保证了非英文内容至少能被正确索引和按字匹配。

## Lex 查询语法

qmd 设计了一套简洁的查询语法（文档见 `docs/SYNTAX.md`），让用户能够精确控制搜索行为：

| 语法 | 含义 | 转换为 FTS5 |
|------|------|------------|
| `term` | 前缀匹配 | `"term"*` |
| `"exact phrase"` | 精确短语 | `"exact phrase"` |
| `-term` | 排除 | `NOT "term"*` |
| `-"phrase"` | 排除短语 | `NOT "phrase"` |

普通词项默认执行前缀匹配，这是一个深思熟虑的选择。用户输入 "config" 时，通常也希望命中 "configuration"、"configure" 等词。前缀匹配与 Porter 词干提取形成互补——Porter 处理后缀变化，前缀匹配处理用户输入不完整的情况。

双引号包裹的短语则执行精确匹配，不附加通配符。这让用户在需要精确搜索时有明确的手段。

## 查询预处理

### sanitizeFTS5Term

`sanitizeFTS5Term()`（L2653–2655）是查询管线的第一道防线。它去除所有非单词字符并转为小写。这一步看似简单，实则至关重要——FTS5 的查询语法中有很多特殊字符（如 `*`、`+`、`:`），如果用户输入中包含这些字符且未被过滤，会导致查询解析失败甚至 SQL 注入风险。

### buildFTS5Query

`buildFTS5Query()`（L2672–2738）是查询转换的核心。它将 Lex 语法解析为 FTS5 能理解的查询字符串。

函数内部维护两个数组：`positive[]` 存放正向匹配条件，`negative[]` 存放排除条件。解析过程逐字符扫描输入字符串：

遇到双引号时，提取引号内的完整短语，不添加前缀通配符。遇到减号时，将后续的词项或短语标记为负向。其余情况视为普通词项，清理后添加前缀通配符。

最终的 FTS5 查询按以下规则组装：正向条件之间用 `AND` 连接；如果存在负向条件，用 `NOT` 逐一追加。

这里有一个关键的边界情况处理：如果查询中只包含负向条件（例如 `-config`），函数返回 `null`。这是因为 FTS5 的 `NOT` 是二元运算符，它必须有一个左操作数。纯否定查询在 FTS5 中无法表达——你不能说"给我所有不包含 config 的文档"而不先指定一个正向条件。

### 查询验证

`validateLexQuery()`（L2752–2761）检查查询字符串中是否包含换行符或未闭合的引号。换行符可能破坏 SQL 语句的结构，未闭合引号则会导致解析器进入错误状态。

`validateSemanticQuery()`（L2744–2749）用于向量搜索和 HyDE 搜索模式。它拒绝包含否定语法的查询，因为向量相似度计算没有"排除"的概念——你无法在嵌入空间中表达"远离这个向量"。

## searchFTS 主函数

`searchFTS()`（L2763–2814）是 BM25 搜索的入口。它编排了从查询解析到结果返回的完整流程。

### 查询构建

函数首先调用 `buildFTS5Query()` 将用户输入转换为 FTS5 查询。如果返回 `null`（纯否定查询），直接返回空结果。

### SQL 查询与权重

核心的 SQL 查询通过 JOIN 将 FTS5 虚拟表、文档元数据表和内容表关联起来：

```sql
SELECT ...
FROM documents_fts AS fts
JOIN documents AS d ON d.filepath = fts.filepath
JOIN content AS c ON c.filepath = d.filepath
WHERE documents_fts MATCH ?
  AND d.collection = ?
ORDER BY bm25(documents_fts, 10.0, 1.0) ASC
```

`bm25()` 函数的权重参数揭示了一个重要的设计决策：`filepath` 字段权重 10.0，`title` 字段权重 1.0，`body` 字段使用默认权重。

文件路径获得 10 倍加权，这背后的逻辑是：当用户搜索 "store" 时，名为 `store.ts` 的文件几乎一定是最相关的结果。路径匹配是极强的信号——用户往往记得文件名但不记得具体内容。标题次之，正文最低。这种加权方案在代码库搜索场景下非常合理。

集合过滤通过可选的 `WHERE d.collection = ?` 子句实现，允许用户将搜索范围限定在特定的文档集合内。

### 分数归一化

BM25 算法返回的原始分数是负数（SQLite FTS5 的惯例），且数值范围不固定——取决于文档集合的大小、词频分布等因素。qmd 使用了一个优雅的归一化函数将其映射到 [0, 1) 区间：

```
normalizedScore = |x| / (1 + |x|)
```

这个函数有几个重要性质：

第一，它是单调递增的。BM25 原始分数的绝对值越大，归一化后的分数越高，排序不变。

第二，它的值域是 [0, 1)，永远不会达到 1。实际效果如下：

| BM25 原始分 | 归一化分数 | 匹配强度 |
|------------|----------|---------|
| -10 | 0.91 | 强匹配 |
| -2 | 0.67 | 中等匹配 |
| -0.5 | 0.33 | 弱匹配 |
| -0.1 | 0.09 | 极弱匹配 |

第三，也是最关键的——它不依赖查询上下文。传统的 min-max 归一化需要知道当前查询结果中的最大和最小分数，这意味着不同查询之间的分数不可比较。而 `|x|/(1+|x|)` 是一个全局函数，任何查询、任何时间返回的分数都在同一尺度上。

### 为什么跨模态可比性如此重要？

qmd 支持多种搜索模态：BM25 全文搜索、向量语义搜索、HyDE 搜索。在融合这些模态的结果时，需要对不同来源的分数进行加权合并。如果 BM25 返回的是 [-∞, 0] 区间的原始分数，而向量搜索返回 [0, 1] 的余弦相似度，两者根本无法直接比较。

`|x|/(1+|x|)` 将 BM25 分数映射到与余弦相似度相同的 [0, 1) 区间，使得跨模态的分数融合变得简单而可靠。不需要每次查询都计算 min/max，不需要维护全局统计信息，只需要一个无状态的数学函数。

## 查询语法的局限性

当前的 Lex 语法有意保持简洁，但也因此存在一些限制。它不支持 OR 逻辑——所有正向条件都是 AND 关系。它不支持字段限定搜索（如 `title:config`）。它也不支持通配符的灵活放置（如 `*config*`）。

这些限制是有意为之的。qmd 面向的是代码文档搜索场景，用户通常输入的是关键词或函数名，复杂的布尔逻辑很少用到。保持语法简洁意味着更少的解析错误、更低的学习成本，以及更可预测的搜索行为。

## 本章小结

qmd 的 BM25 搜索系统展现了"简洁但不简单"的设计哲学。FTS5 虚拟表利用 Porter 词干提取和 Unicode61 分词器覆盖了多语言场景；Lex 查询语法在易用性和表达力之间找到了平衡点；`buildFTS5Query()` 正确处理了前缀匹配、短语搜索和否定查询的各种边界情况；路径字段的 10 倍加权体现了对代码库搜索场景的深刻理解；而 `|x|/(1+|x|)` 归一化函数则以极低的复杂度解决了跨模态分数可比性这一关键问题。整套系统没有使用任何外部搜索引擎依赖，仅凭 SQLite 内置的 FTS5 就实现了生产级的全文搜索能力。
