# 第 9 章：LLM 抽象层

> 一个本地文档搜索工具需要三种不同的 AI 能力：嵌入、生成、重排。如果每个功能都独立管理模型的加载和释放，代码会迅速变成一团乱麻。`src/llm.ts` 用 1546 行代码构建了一个统一的 LLM 抽象层，将 node-llama-cpp 的复杂性封装在清晰的接口之后。这一章我们将拆解这个抽象层的设计——从 LlamaCpp 类的模型管理，到 ILLMSession 的会话机制，再到查询扩展和重排的实际应用。

## LlamaCpp 类：核心抽象

### 单一入口的设计

`LlamaCpp` 类是整个 LLM 层的中枢。它通过 `getLlama()` 创建底层的 Llama 实例，然后在此基础上管理三个模型的生命周期：嵌入模型、生成模型、重排模型。三种模型共享同一个 Llama 运行时，这意味着 GPU 资源的分配和调度可以在一个地方统一控制。

这个设计的关键价值在于：上层代码不需要知道 node-llama-cpp 的任何细节。无论是 `src/store.ts` 中的批量嵌入，还是搜索流程中的查询扩展，它们看到的都是简洁的方法调用，而不是 GGUF 文件的加载参数或 GPU 内存的分配策略。

### 模型 URI 与 HuggingFace 解析

qmd 使用统一的 URI 格式引用模型文件：`hf:org/repo/file.gguf`。这个格式指向 HuggingFace 仓库中的具体 GGUF 文件。`resolveModelFile()` 函数负责将这个 URI 解析为本地文件路径——如果模型尚未下载，它会自动从 HuggingFace 拉取。

三个默认模型的 URI 分别对应：

- `DEFAULT_EMBED_MODEL_URI`：embeddinggemma 的 GGUF 文件，用于文本嵌入。
- `DEFAULT_GENERATE_MODEL_URI`：Qwen3-1.7B 的 GGUF 文件，用于查询扩展。
- `DEFAULT_RERANK_MODEL_URI`：qwen3-reranker 0.6B Q8_0 的 GGUF 文件，用于结果重排。

所有模型下载后缓存在 `~/.cache/llama-cpp/models/` 目录下，后续使用直接从本地加载，无需重复下载。GGUF 格式是 llama.cpp 生态的标准格式，针对本地推理做了深度优化。

### 不活跃超时机制

模型加载到 GPU 显存中是昂贵的操作，但长期占用显存又会阻碍其他应用。`LlamaCpp` 类实现了不活跃超时（inactivity timeout）机制：当一个模型在指定时间内没有被使用，系统会自动释放它占用的资源。

超时时间根据使用场景不同而有所区别。在 SDK 模式下默认为 5 分钟——这适合程序化调用的场景，模型用完即释放，避免占用开发机的宝贵显存。在 CLI 模式下默认为 30 分钟——这适合交互式使用的场景，用户可能在两次搜索之间做其他事情，较长的超时避免了频繁的模型加载延迟。

这个机制在代码层面通过计时器实现。每次模型被调用时重置计时器，计时器到期后调用 dispose 释放资源。下一次需要该模型时再重新加载。

## ILLMSession 接口与会话管理

### 接口定义

`ILLMSession` 是 LLM 层暴露给外部的核心接口，定义了三个基本操作：

- `embed(text)` —— 将单条文本转为嵌入向量，返回 `number[]`。
- `embedBatch(texts)` —— 将多条文本批量转为嵌入向量，返回 `number[][]`。批量操作显著高效于多次单条调用，因为它能让 GPU 在一次前向传播中处理多个输入。
- `generate(prompt)` —— 根据 prompt 生成文本，用于查询扩展等场景。

接口设计刻意保持了最小化。它不暴露模型参数、采样策略、上下文窗口等底层细节，让调用方专注于业务逻辑。如果未来需要替换 node-llama-cpp 为其他推理后端（比如 ONNX Runtime 或远程 API），只需实现这三个方法即可。

### withLLMSession 与 withLLMSessionForLlm

会话管理是 LLM 层中一个容易被忽视但极其重要的部分。`withLLMSession` 和 `withLLMSessionForLlm` 是两个会话管理函数，采用了经典的 "借用-归还" 模式：

```typescript
await withLLMSession(async (session) => {
  const embedding = await session.embed(text);
  // 使用 embedding...
});
// 会话在这里被自动清理
```

这个模式保证了两件事。第一，会话结束后资源被正确释放，不会出现内存泄漏。第二，会话期间模型的不活跃计时器被暂停或重置，不会在使用过程中被意外卸载。

`withLLMSessionForLlm` 是 `withLLMSession` 的变体，允许调用方指定特定的 Llama 实例。这在测试场景中尤其有用，可以注入 mock 对象来替代真实的模型。

## 查询扩展：expandQuery()

### 从自然语言到结构化子查询

`expandQuery()` 是 LLM 层中最有趣的功能之一。它利用生成模型将用户的自然语言查询扩展为多个类型化的子查询。输出是一个 `ExpandedQuery[]` 数组，每个元素包含查询文本和查询类型。

查询类型包括三种：

- **lex**（词法查询）：适合全文搜索引擎的关键词组合，比如将 "如何配置数据库连接" 扩展为 "database connection config setup"。
- **vec**（向量查询）：适合语义搜索的自然语言描述，保留了原始查询的语义完整性。
- **hyde**（假设性文档）：Hypothetical Document Embedding 的缩写，生成一段 "假想的理想答案"，然后用这段文本去做向量搜索。这个技巧来自 2022 年的同名论文，核心思路是：一个好的答案与相关文档的语义距离，往往比问题本身与相关文档的距离更近。

### 结构化输出的解析

`expandQuery()` 使用 Qwen3-1.7B 生成结构化的输出文本，然后将其解析为 `ExpandedQuery[]` 对象。选择 1.7B 这样的小模型做查询扩展是一个务实的决定：查询扩展不需要深度推理能力，一个轻量级模型就足以胜任，而且响应速度更快，对用户体验的影响更小。

## 重排机制

### 交叉编码器的角色

向量搜索返回的结果按余弦相似度排序，但这个排序并不总是最优的。嵌入模型是独立编码查询和文档的（bi-encoder），无法捕捉查询与文档之间的细粒度交互。重排模型使用交叉编码器（cross-encoder）架构，同时接收查询和文档作为输入，能够更准确地判断相关性。

`LlamaCpp` 类提供了 `rerank(query, documents, options)` 方法，返回每个文档的相关性分数。重排通常只对向量搜索返回的 top-K 结果执行，因此计算量可控。

### 重排模型的选择

默认的重排模型是 `ExpedientFalcon/qwen3-reranker:0.6b-q8_0`。0.6B 参数量意味着它可以在几乎所有现代硬件上快速运行。Q8_0 量化在精度损失极小的前提下显著减少了模型体积和显存占用。这个选择再次体现了 qmd "本地优先、轻量为王" 的设计哲学。

## 设备感知与硬件适配

### GPU 检测与资源管理

node-llama-cpp 在底层处理了 GPU 的检测和调度。它支持三种 GPU 后端：macOS 上的 Metal、NVIDIA 的 CUDA、以及跨平台的 Vulkan。`LlamaCpp` 类通过设备信息接口获取当前硬件状态：GPU 名称、显存（VRAM）大小、层卸载（offloading）状态、CPU 核心数等。

层卸载是一个关键概念：当模型的全部层无法放入 GPU 显存时，部分层会被保留在 CPU 内存中执行。这虽然比全 GPU 推理慢，但确保了在显存不足的设备上也能运行。qmd 不强制要求高端 GPU，这是 "本地优先" 理念的具体体现。

### GGUF 格式的优势

所有模型都使用 GGUF（GPT-Generated Unified Format）格式存储。相比原始的 PyTorch 或 SafeTensors 权重文件，GGUF 有几个关键优势：支持多种量化级别（Q4、Q5、Q8 等）以适配不同硬件；元数据内嵌在文件中，无需额外的配置文件；mmap 加载支持让模型可以直接从磁盘映射到内存，启动速度更快。

## 设计哲学：完全本地化

### 无需 API Key 的 AI

qmd 的 LLM 层最突出的设计原则是完全本地化。没有 API key，没有云服务调用，没有网络延迟。所有推理都在用户自己的机器上完成。这意味着：文档内容永远不会离开本地设备，搜索速度不受网络波动影响，离线环境下也能正常工作。

### 三模型一抽象

管理三个不同用途的模型是一个不小的工程挑战。嵌入模型需要高吞吐的批量处理，生成模型需要灵活的采样控制，重排模型需要高效的成对评分。`LlamaCpp` 类将这三种截然不同的推理模式统一在一个抽象之后，通过不活跃超时机制动态管理模型的加载与释放。

这种设计的好处在于资源效率。在实际使用中，三个模型很少需要同时驻留在显存中——嵌入和重排通常在索引阶段使用，查询扩展在搜索阶段使用。超时机制让不活跃的模型自动释放，为活跃模型腾出空间。

### Qwen3 嵌入模型的特殊处理

`isQwen3EmbeddingModel()` 函数（第 29-31 行）是一个值得注意的小细节。它通过正则表达式检测模型名称中是否包含 Qwen3 Embedding 的特征字符串。一旦检测为 Qwen3 系列，`formatQueryForEmbedding()` 会切换到专用的 prompt 格式，使用 `Instruct:` 和 `Query:` 前缀替代默认的 `task:` 格式。

这种按模型类型分发行为的模式在整个 LLM 层中反复出现。它让 qmd 能够支持多种模型家族，同时保持统一的上层 API。用户只需要更换模型名称，底层的 prompt 格式、上下文长度等参数会自动适配。

## 本章小结

本章拆解了 `src/llm.ts` 中 LLM 抽象层的完整架构。`LlamaCpp` 类作为核心入口，统一管理嵌入、生成、重排三个模型的生命周期，通过 HuggingFace URI 解析实现模型的自动下载与缓存，通过不活跃超时机制实现显存的动态调度。`ILLMSession` 接口以最小化的 API 表面暴露嵌入和生成能力，`withLLMSession` 确保资源的正确清理。在功能层面，`expandQuery()` 利用小型生成模型将自然语言查询扩展为词法、向量、假设性文档三种子查询，交叉编码器重排模型则对候选结果做精细化排序。整个系统完全运行在本地，无需网络连接和 API 密钥，三种模型通过一个抽象层协同工作，在消费级硬件上实现了专业级的语义检索能力。
