# 第 10 章：混合查询管线

> `hybridQuery()` 是 qmd 检索系统的心脏——一条八步管线将 BM25 全文检索、向量语义搜索、LLM 查询扩展和交叉编码器重排熔于一炉，在延迟与召回之间走出了一条务实的中间路线。

## 管线全景

<!-- 文件: src/store.ts L3712-3990 -->

`hybridQuery()` 函数跨越近 280 行，是 `src/store.ts` 中最长的单一函数。它接收用户原始查询、可选的 `intent` 字符串以及各种限制参数，返回 `HybridQueryResult[]`。整条管线可以拆解为八个有序阶段：BM25 探针、查询扩展、分型路由搜索、RRF 融合、分块选择、重排、位置感知混合、去重过滤。

每个阶段都有明确的短路或跳过条件，这使得简单查询可以在几毫秒内返回，而模糊查询则自动启用全链路。理解这八步之间的数据流动，就等于理解了 qmd 检索的全部核心逻辑。

## 第一步：BM25 探针

### 强信号检测

<!-- 文件: src/store.ts L3738-3743 -->

管线的第一步不是盲目展开搜索，而是先用 FTS（全文搜索）做一次快速试探。`hybridQuery()` 在第 3738 行发起一次标准 BM25 查询，拿到初始结果列表后，检查两个常量阈值：

<!-- 文件: src/store.ts L226-232 -->

```ts
// src/store.ts L226-232
const STRONG_SIGNAL_MIN_SCORE = 0.85;
const STRONG_SIGNAL_MIN_GAP = 0.15;
```

判断逻辑非常直觉：如果第一名的 BM25 得分 ≥ 0.85，**且**与第二名的分差 ≥ 0.15，则认为这是一个"强信号"。强信号意味着 FTS 已经高度自信地找到了目标文档，后续昂贵的 LLM 扩展和向量搜索可以安全跳过。

### 为什么 intent 会禁用旁路

这里有一个关键的设计细节：当调用方传入了 `intent` 参数时，即使 BM25 探针检测到强信号，旁路也会被**禁用**。原因在于，带 intent 的查询往往是模糊的——用户说"性能"，intent 说"网页加载时间"，BM25 可能匹配到体育成绩相关的文档，表面得分很高但语义完全错误。此时必须让向量路径参与纠偏。

如果命中强信号且无 intent，`SearchHooks.onStrongSignal` 回调被触发（第 3741 行），CLI 可以据此输出提示信息，让用户知道搜索走了快速路径。

## 第二步：查询扩展

<!-- 文件: src/store.ts L3748-3754 -->

当强信号未被检测到（或被 intent 禁用）时，管线进入 `expandQuery()` 调用。这一步通过 LLM 生成多个查询变体，分为三种类型：`lex`（关键词变体）、`vec`（语义变体）和 `hyde`（假设性文档）。

扩展的细节将在第 11 章详述。这里需要注意的是，`expandQuery()` 本身带有缓存机制，相同的查询 + 模型 + intent 组合不会重复调用 LLM。扩展结果会被过滤，去掉与原始查询完全相同的变体，避免重复搜索。

`SearchHooks.onExpandStart` 和 `onExpand` 回调分别在扩展开始和完成时触发，CLI 用它们来显示"正在扩展查询..."之类的进度提示。

## 第三步：分型路由搜索

<!-- 文件: src/store.ts L3773-3828 -->

这一步是管线中计算量最大的环节，也是 qmd 混合检索"混合"二字的核心所在。所有扩展结果根据类型被路由到不同的搜索引擎：

| 扩展类型 | 目标引擎 | 执行方式 |
|---------|---------|---------|
| `lex` | FTS 全文搜索 | 同步执行，毫秒级 |
| `vec` | sqlite-vec 向量搜索 | 批量嵌入后查询 |
| `hyde` | sqlite-vec 向量搜索 | 批量嵌入后查询 |

关键性能决策在于**批量嵌入**：原始查询与所有 `vec`、`hyde` 变体被收集到一个数组中，通过 `embedBatch()` 一次性发送给嵌入模型。这比逐条嵌入快得多——假设有 4 个向量变体，批量方式只需一次网络往返，而逐条方式需要 4 次。

FTS 搜索在本地 SQLite 上执行，延迟在个位数毫秒级别，所以可以同步完成。向量搜索依赖 sqlite-vec 扩展，使用预先计算好的文档嵌入向量做近似最近邻查找。

## 第四步：RRF 融合

<!-- 文件: src/store.ts L3831-3834 -->

多路搜索产生多个排序列表，需要一种方式将它们合并。qmd 使用的是经典的 **Reciprocal Rank Fusion**（RRF）算法，实现在 `reciprocalRankFusion()` 函数中。

### reciprocalRankFusion() 的实现

<!-- 文件: src/store.ts L3056-3098 -->

该函数接收一个带权重的列表数组 `Array<{results, weight}>`，内部使用 `Map<string, { result, rrfScore, topRank }>` 做累积。对每个列表中排名为 `rank` 的文档，累加公式为：

```
score += weight / (k + rank + 1)
```

其中 `k = 60` 是标准 RRF 常数。权重的设置很有讲究：前两个列表（通常是原始查询的 FTS 和向量结果）获得 **2 倍权重**，因为原始查询本身最能代表用户意图，扩展变体只是补充。

此外还有一个**顶部排名加分**机制：如果某个文档在任何一个列表中排名第 1，额外加 0.05 分；排名第 2 或第 3，额外加 0.02 分。这奖励了那些在至少一个检索维度上表现突出的文档，使它们不容易被平均化淹没。

## 第五步：分块选择

<!-- 文件: src/store.ts L3840-3862 -->

这一步是管线中最精巧的性能优化。RRF 融合后我们有了候选文档列表，但直接把完整文档体送去重排的代价是 **O(tokens)**——一篇 5000 字的文档会消耗大量 reranker 计算。

qmd 的解决方案是：对每个候选文档，调用 `chunkDocument()` 将正文切成若干块，然后用关键词重叠度选出**最佳块**。重叠度的计算结合了两种来源：

- 查询本身的分词（权重 1.0）
- intent 中提取的术语（权重 0.5，通过 `INTENT_WEIGHT_CHUNK` 常量控制）

intent 术语的权重设为 0.5 而非 1.0，是因为 intent 是对查询方向的补充说明，不应完全主导块的选择。最终每个候选文档只保留一个最佳块（`bestChunk`）及其在文档中的位置（`bestChunkPos`），送入下一步。

这个"先粗选文档，再精选块"的两级策略，是 qmd 检索延迟能控制在可接受范围内的关键。`RERANK_CANDIDATE_LIMIT = 40`（第 232 行）进一步限制了进入块选择的候选数量。

## 第六步：重排

<!-- 文件: src/store.ts L3914-3926 -->

最佳块（而非完整文档体）被送入 `rerank()` 函数。重排使用交叉编码器模型，能够理解查询与文档之间的深层语义关系，弥补 BM25 和向量检索在精细排序上的不足。

重排只处理块文本，这意味着即使有 40 个候选，每个块平均 200-300 token，总输入也只有 8000-12000 token——远小于直接重排完整文档的开销。`rerank()` 的具体实现将在第 11 章展开。

## 第七步：位置感知混合

<!-- 文件: src/store.ts L3929-3978 -->

这一步是 qmd 检索管线中最具原创性的设计。重排完成后，需要将 RRF 分数和 reranker 分数融合成最终得分。朴素做法是固定权重线性加权，但 qmd 采用了**根据 RRF 排名动态调整权重**的策略：

| RRF 排名 | rrfWeight |
|----------|-----------|
| 前 3 名 | 0.75 |
| 第 4-10 名 | 0.60 |
| 第 11 名及以后 | 0.40 |

最终混合公式为：

```
blendedScore = rrfWeight * (1 / rrfRank) + (1 - rrfWeight) * rerankScore
```

### 设计动机

为什么排名靠前的文档要给 RRF 更高的权重？因为 RRF 排名靠前意味着该文档在**多个检索维度**上都表现良好，这种一致性本身就是强烈的相关性信号。如果 reranker 对这些文档给出低分，更可能是 reranker 的片面判断（比如块选择不理想），而非文档真的不相关。

相反，RRF 排名靠后的文档只在少数维度上被召回，此时应该更信任 reranker 的精细判断，所以 reranker 权重升高到 0.60。这种"位置感知"策略有效地保护了多路一致的好结果，同时给 reranker 在长尾区域留出了足够的调整空间。

## 第八步：去重与过滤

<!-- 文件: src/store.ts L3981-3989 -->

最后一步相对简单：按文件路径去重（同一文件只保留得分最高的条目），应用 `minScore` 阈值过滤低分结果，然后按 `limit` 参数截取最终返回数量。

## 返回类型与进度钩子

### HybridQueryResult

<!-- 文件: src/store.ts L3680-3691 -->

返回类型包含了检索全链路的关键信息：`file`（文件路径）、`displayPath`（显示路径）、`title`、`body`（完整文档体）、`bestChunk`（最佳块文本）、`bestChunkPos`（块在文档中的起止位置）、`score`（最终混合得分）、`context`（上下文信息）、`docid`，以及可选的 `explain` 追踪字段用于调试。

### SearchHooks

<!-- 文件: src/store.ts L3652-3667 -->

`SearchHooks` 接口定义了管线各阶段的进度回调：`onStrongSignal`、`onExpandStart`、`onExpand`、`onEmbedStart`、`onEmbedDone`、`onRerankStart`、`onRerankDone`。这些回调让 CLI 能够在长时间运行的搜索过程中实时展示进度，而不是让用户面对一片空白等待。

这种 hooks 设计将 I/O 表现与核心逻辑解耦——`hybridQuery()` 本身不关心进度如何展示，它只在合适的时机调用回调函数，具体的 UI 行为由调用方决定。

## 本章小结

`hybridQuery()` 的八步管线体现了一种"分层递进"的检索哲学：先用最便宜的 BM25 试探，能短路就短路；不能短路则用 LLM 扩展查询空间，再通过分型路由最大化召回；RRF 融合将多路结果统一排序；分块选择把重排开销从 O(文档) 降到 O(块)；位置感知混合在 RRF 的全局视野与 reranker 的局部精度之间取得平衡。每一步都有明确的性能理由，每一个阈值都服务于延迟与质量的权衡。理解了这条管线，就掌握了 qmd 检索系统的核心脉络。
