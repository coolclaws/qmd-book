# 第 11 章：查询扩展与重排

> 如果说混合查询管线是 qmd 检索的骨架，那么查询扩展和重排就是它的神经系统——前者负责"想到更多可能"，后者负责"判断哪个最好"。两者都依赖 LLM，也都通过精心设计的缓存策略将 LLM 调用降到最低。

## 查询扩展：expandQuery()

### 函数签名与流程

<!-- 文件: src/store.ts L2968-3001 -->

`expandQuery()` 是查询扩展的入口函数，位于 `src/store.ts` 第 2968 行。它接收原始查询字符串和可选的 intent，返回 `ExpandedQuery[]` 数组。整个函数不到 40 行，但流程设计非常紧凑。

第一步是**查缓存**。函数调用 `getCacheKey("expandQuery", {query, model, intent})` 生成缓存键，然后在 `llm_cache` 表中查找。只有缓存未命中时，才会调用 LLM。这一设计对用户体验影响巨大：同一个查询在首次搜索后，后续搜索的扩展步骤延迟从数百毫秒降至零。

第二步处理了一个向后兼容细节：旧版缓存格式中扩展结果存储在 `text` 字段，新版使用 `query` 字段。代码中有显式的格式迁移逻辑，确保老缓存不会导致运行时错误。

第三步调用 `llm.expandQuery(query, {intent})`，由底层的生成模型产出扩展变体。最后一步过滤掉与原始查询完全相同的变体——LLM 偶尔会把原始查询原样返回，这种重复没有意义。

### ExpandedQuery 类型

<!-- 文件: src/store.ts L242-247 -->

```ts
// src/store.ts L242-247
type ExpandedQuery = {
  type: 'lex' | 'vec' | 'hyde';
  query: string;
};
```

三种扩展类型各有分工：

| 类型 | 含义 | 路由目标 | 典型示例 |
|------|------|---------|---------|
| `lex` | 关键词变体 | FTS 全文搜索 | 原查询"配置文件"→ "config yaml toml" |
| `vec` | 语义变体 | sqlite-vec 向量搜索 | 原查询"如何部署"→ "deployment instructions setup" |
| `hyde` | 假设性文档 | sqlite-vec 向量搜索 | 原查询"错误处理"→ "The error handler catches exceptions and logs them to stderr..." |

`lex` 类型只走 FTS，因为关键词变体的价值在于匹配文档中的精确词汇，向量化反而可能模糊焦点。`vec` 和 `hyde` 只走向量搜索，因为它们的价值在于语义相似度，BM25 无法理解同义词和释义。

`hyde`（Hypothetical Document Embeddings）是一种巧妙的技术：让 LLM 生成一段"假设性答案文档"，然后用这段文档的嵌入去搜索真实文档。由于假设性文档在语义空间中更接近真正的答案文档，这种方式往往能召回纯关键词搜索遗漏的结果。

## 强信号检测的阈值设计

<!-- 文件: src/store.ts L226-232 -->

在第 10 章中我们介绍了强信号检测的存在，这里深入讨论阈值的选择理由。

```ts
// src/store.ts L226-232
const STRONG_SIGNAL_MIN_SCORE = 0.85;
const STRONG_SIGNAL_MIN_GAP = 0.15;
const RERANK_CANDIDATE_LIMIT = 40;
```

0.85 的最低分数阈值意味着 BM25 必须对首选结果非常自信。在实践中，BM25 分数达到 0.85 通常意味着查询词与文档标题或正文有高度的精确匹配。0.15 的最小分差则确保首选结果不只是"好"，而是"明显好于第二名"。

这两个阈值共同构成了一个保守的短路条件。宁可多做一次 LLM 扩展（代价是几百毫秒延迟），也不愿错过相关文档。只有当证据压倒性地指向一个结果时，才跳过扩展。

`RERANK_CANDIDATE_LIMIT = 40` 控制进入重排阶段的最大候选数。这个数字是延迟与召回的折中：太少会丢失长尾相关文档，太多会让重排阶段的 LLM 调用变得昂贵。40 在大多数文档库规模下是一个合理的上界。

## 重排：rerank() 函数

### 核心逻辑

<!-- 文件: src/store.ts L3007-3050 -->

`rerank()` 函数位于第 3007 行，负责对候选块进行交叉编码器重排。它的设计有三个值得注意的特点。

**第一，intent 前缀注入。** 如果调用方传入了 intent，`rerank()` 会将 intent 拼接到查询前面作为重排查询。这给了 reranker 额外的领域上下文。例如，查询"performance"加上 intent "web page load times"，reranker 就知道应该优先给 Web 性能相关的块打高分，而非体育成绩相关的块。

**第二，基于内容的缓存。** 每个文档的缓存键由查询文本 + 块文本共同决定，**不包含文件路径**。这意味着如果两个不同路径的文件恰好有相同的内容块，重排分数可以复用。这种设计特别适合文档库中存在副本或模板文件的场景。

代码中还保留了对旧版缓存键的兼容查找（legacy cache key fallback）。当新格式的缓存键未命中时，会尝试旧格式，确保用户升级后不需要重建缓存。

**第三，批量重排未缓存的文档。** 函数先遍历所有候选，收集缓存未命中的文档，然后一次性调用 `llm.rerank()` 批量处理。这避免了逐个调用 reranker 的网络开销。处理完成后，新结果被写入缓存，最终按分数降序排列返回。

### rerank 与 hybridQuery 的协作

在第 10 章的管线中，`rerank()` 处理的是经过分块选择后的最佳块。这意味着 reranker 看到的文本长度是可控的——通常在 200-500 token 之间，而非整篇文档可能的数千 token。这是一个至关重要的架构决策：它让 qmd 可以使用高质量但计算昂贵的交叉编码器，而不用担心 token 数爆炸。

## 缓存策略的整体设计

### getCacheKey() 与存储机制

<!-- 文件: src/store.ts L1763-1768 -->

所有 LLM 相关的缓存都通过 `getCacheKey()` 函数生成键。该函数将 URL 和请求体 JSON 拼接后计算 SHA-256 哈希。使用哈希而非原始文本作为键，既节省存储空间，也避免了长文本作为数据库键的性能问题。

`getCachedResult()` 和 `setCachedResult()` 是对 `llm_cache` 表的简单读写封装。缓存表采用 key-value 结构，值为 JSON 序列化的结果。

### 概率性清理

<!-- 文件: src/store.ts L1778-1779 -->

缓存不会无限增长。每次写入时，有 **1% 的概率**触发一次清理操作，将缓存表裁剪到最近的 1000 条记录。这种概率性策略避免了每次写入都检查缓存大小的开销，同时在统计上保证缓存不会膨胀失控。

`clearCache()` 函数则提供了完全清空缓存的能力，执行 `DELETE FROM llm_cache`。这在用户切换了 LLM 模型或嵌入模型后非常有用，因为旧缓存在新模型下可能产生错误结果。

## structuredSearch()：另一个入口

<!-- 文件: src/store.ts -->

`structuredSearch()` 是面向 SDK 和 MCP 协议的查询入口。与 `hybridQuery()` 的区别在于：它接收的是**已经预分类的查询**，即调用方已经明确标注了每个查询属于 `lex`、`vec` 还是 `hyde` 类型。

因此，`structuredSearch()` 跳过了强信号检测和 `expandQuery()` 两个阶段，直接进入分型路由搜索。之后的 RRF 融合、分块选择、重排、位置感知混合和去重过滤与 `hybridQuery()` 完全一致。

这种双入口设计允许上层应用根据自身能力选择合适的接口：CLI 和简单客户端使用 `hybridQuery()` 享受全自动的扩展流程；SDK 集成方可以自行控制查询扩展策略，通过 `structuredSearch()` 直接注入分类后的查询。

## Intent 系统的全链路影响

### intent 如何贯穿五个阶段

intent 不是一个简单的"附加参数"，它深入影响了管线的五个阶段：

1. **查询扩展**：intent 被传递给 `llm.expandQuery()`，引导 LLM 生成更有方向性的变体
2. **强信号旁路**：有 intent 时禁用旁路，因为 BM25 无法理解语义方向
3. **分块选择**：intent 术语以 0.5 权重（`INTENT_WEIGHT_CHUNK`）参与关键词重叠计算
4. **重排**：intent 被拼接到重排查询前面，为 reranker 提供领域上下文
5. **片段提取**：intent 术语以 0.3 权重（`INTENT_WEIGHT_SNIPPET`）影响最终展示的文本片段选择

### extractIntentTerms() 的实现

<!-- 文件: src/store.ts L3552-3556 -->

`extractIntentTerms()` 函数将 intent 字符串拆分为可用于匹配的术语列表。处理流程很简洁：按空白符分词，去除标点符号，过滤掉停用词（`INTENT_STOP_WORDS` 集合中的常见词如 "the"、"a"、"is" 等）。

这种轻量级的预处理确保 intent 术语不会被虚词稀释。例如 intent 为 "web page load times" 时，提取出的有效术语是 "web"、"page"、"load"、"times"——每一个都携带搜索相关的语义信息。

### 权重的差异化设计

intent 在不同阶段使用不同权重是一个深思熟虑的选择。在分块选择阶段权重为 0.5，因为块选择需要较强的 intent 引导来避免选中无关段落。在片段提取阶段权重降至 0.3，因为片段主要应呈现查询本身的匹配上下文，intent 只起微调作用。

这种差异化权重体现了 qmd 的设计哲学：intent 是方向盘而非油门，它引导搜索朝正确的方向走，但不应喧宾夺主，压过用户查询本身的信号。

## 本章小结

查询扩展和重排是 qmd 混合管线中两个最依赖 LLM 的环节。`expandQuery()` 通过生成 `lex`、`vec`、`hyde` 三种类型的变体，将单一查询扩展为多维搜索空间；`rerank()` 通过交叉编码器在精选的块上做精细排序，弥补统计检索的语义盲区。两者都采用了激进的缓存策略——扩展缓存以查询 + 模型 + intent 为键，重排缓存以查询 + 块内容为键，概率性清理避免缓存膨胀。intent 系统贯穿扩展、块选择、重排和片段提取四个阶段，通过差异化权重在不同环节发挥恰到好处的引导作用。`structuredSearch()` 作为第二入口，为 SDK 集成方提供了跳过自动扩展的能力，体现了接口设计中"简单场景自动化、高级场景可控化"的原则。
