## 第 13 章：MCP 服务器

> Model Context Protocol（MCP）是 Anthropic 提出的大模型工具调用标准，而 qmd 的 MCP 服务器实现是将本地代码索引能力暴露给 AI 助手的桥梁。`src/mcp/server.ts` 全文 807 行，实现了 stdio 和 HTTP 两种传输协议、四个搜索与检索工具、一个资源模板以及动态系统提示生成。本章将逐层剖析这个服务器的架构设计与实现细节。

### 两种传输协议

qmd 的 MCP 服务器支持两种传输方式，分别面向不同的集成场景。

第一种是 stdio 传输，由 `startMcpServer()` 函数实现（第 522 至 527 行）。这是默认模式，专为 Claude Desktop 和 Claude Code 等桌面端 AI 助手设计。在这种模式下，qmd 作为子进程启动，通过标准输入输出与宿主进程通信。消息采用 JSON-RPC 格式，每条消息以换行符分隔。stdio 模式的优势在于零配置——不需要端口、不需要认证、不需要网络，进程间通信天然安全。

第二种是 HTTP 传输，由 `startMcpHttpServer()` 函数实现（第 543 至 802 行）。这种模式使用 Streamable HTTP 协议，返回 JSON 响应。HTTP 模式面向更广泛的集成场景：Web 应用、远程开发环境、多客户端并发访问等。

HTTP 服务器的实现有几个值得关注的细节。首先是会话管理：服务器维护一个 `Map<string, WebStandardStreamableHTTPServerTransport>` 映射表，为每个客户端分配独立的 `McpServer` 和 `Transport` 实例对。这是 MCP 规范的要求——每个会话必须拥有独立的状态上下文。其次，底层的 qmd store 在所有会话之间共享，因为 SQLite 对并发读取是安全的，不需要为每个会话创建独立的数据库连接。

HTTP 服务器还提供了两个 REST 快捷端点。`POST /query` 和 `POST /search` 绕过了 MCP 协议的 JSON-RPC 包装，允许简单的 HTTP 客户端直接发起搜索请求。这对于不需要完整 MCP 能力的轻量级集成非常实用。此外，`GET /health` 端点返回服务器健康状态，适用于监控和负载均衡场景。服务器对每个请求都记录时间戳和工具描述信息，方便调试和运维。

### 四个 MCP 工具

MCP 服务器注册了四个工具，覆盖了搜索和检索的核心场景。

**query 工具**（第 224 至 341 行）是主搜索工具，也是 AI 助手最常调用的接口。它接收一个 `searches[]` 数组作为输入，每个元素是一个子搜索请求，可以指定搜索类型（`lex` 词法搜索、`vec` 向量搜索或 `hyde` 假设文档嵌入）。此外还支持 `limit`（结果数量上限）、`minScore`（最低分数阈值）、`candidateLimit`（候选集大小）、`collections[]`（限定搜索范围的集合列表）和 `intent`（搜索意图描述）等参数。

query 工具的描述文本本身就是一份详尽的使用指南，包含 Markdown 格式的示例和最佳实践。这是有意为之的设计——LLM 通过阅读工具描述来理解如何正确调用工具，因此丰富的描述文本直接提升了 AI 助手的搜索质量。输入参数的校验使用 Zod schema（例如 `subSearchSchema` 通过 `z.object` 定义），在运行时对 LLM 生成的参数进行严格类型检查。

**get 工具**（第 347 至 406 行）用于检索单个文档。输入可以是文件路径或文档 ID，并且支持 `:line` 后缀语法（例如 `foo.md:120`），直接定位到指定行号附近的内容。当请求的文档不存在时，get 工具不会简单返回错误，而是利用索引查找相似路径的文件并在响应中列出，帮助 AI 助手自我修正。

**multi_get 工具**（第 412 至 479 行）支持批量检索。输入可以是 glob 模式（如 `src/**/*.ts`）或逗号分隔的路径列表。为了防止单次请求返回过多数据，该工具会跳过超过 `maxBytes` 限制的大文件。这个工具在 AI 助手需要理解多个相关文件时特别有用，例如分析一个模块的所有源文件。

**status 工具**（第 485 至 513 行）返回索引的健康信息和集合列表。AI 助手可以在搜索前调用此工具，了解当前可搜索的内容范围、索引的新鲜度以及是否存在需要更新的集合。

### 资源模板

除了工具之外，MCP 服务器还注册了一个资源模板（第 172 至 207 行）：`qmd://{+path}`。这是一个 URI 模板，遵循 RFC 6570 规范，`{+path}` 表示路径参数不做 URL 编码。通过这个资源模板，MCP 客户端可以直接以 URI 的方式访问索引中的文档，例如 `qmd://src/index.ts`。

资源访问是只读的，返回文档的完整内容。与 get 工具相比，资源模板提供了一种更符合 MCP 规范的文档访问方式——工具用于执行操作，资源用于读取数据。在实践中，一些 MCP 客户端会优先通过资源 URI 获取文档，仅在需要搜索时才调用工具。

### buildInstructions()：动态系统提示

`buildInstructions()` 函数定义在第 92 至 151 行，是 MCP 服务器中最精巧的设计之一。它根据索引的实际状态动态生成系统提示文本，在 MCP 初始化响应中注入到 LLM 的上下文里。

这个函数首先查询索引状态，获取所有集合的列表及其上下文描述。然后它构造一段结构化的提示文本，包含以下信息：当前可搜索的内容范围、每个集合的名称和上下文描述、索引的能力边界（例如是否已生成向量嵌入）以及搜索技巧和最佳实践。

动态提示的价值在于上下文相关性。一个索引了 Python 项目的 qmd 实例和一个索引了 TypeScript 项目的实例，它们生成的系统提示是完全不同的。AI 助手因此能够根据实际可用的数据调整其搜索策略，而不是依赖通用的固定提示。

例如，如果某个集合设置了上下文"这是一个 React 前端项目，使用 TypeScript 和 Zustand 状态管理"，那么 `buildInstructions()` 会将这段描述包含在系统提示中。AI 助手在后续交互中就能生成更贴合项目实际技术栈的搜索查询。

### HTTP 服务器的会话模型

HTTP 传输模式（第 543 至 802 行）的会话管理值得单独讨论。MCP 规范要求每个客户端会话拥有独立的状态，这意味着服务器不能使用单个全局 `McpServer` 实例来处理所有请求。

qmd 的实现方案是维护一个会话映射表。当新客户端首次连接时，服务器创建一个新的 `WebStandardStreamableHTTPServerTransport` 实例和与之配对的 `McpServer` 实例，将它们存入映射表。后续来自同一客户端的请求通过会话标识符路由到对应的 transport 和 server。

关键的架构决策是 store 共享。所有会话复用同一个 qmd store 实例，因为搜索和检索操作本质上是无状态的——它们不修改索引，只读取数据。SQLite 的并发读取机制保证了线程安全性。这种"共享数据、隔离协议"的设计在保持 MCP 规范合规性的同时，最小化了资源开销。

### Zod Schema 验证

MCP 工具的输入参数通过 Zod schema 进行运行时验证。以 query 工具为例，`subSearchSchema` 使用 `z.object()` 定义了子搜索请求的结构，包括查询文本的类型（字符串）、搜索模式的枚举值（`lex`、`vec`、`hyde`）以及各参数的可选性和默认值。

Zod 验证在 MCP 场景下尤为重要，因为工具的调用者是 LLM 而非人类开发者。LLM 生成的 JSON 参数可能包含类型错误、缺失字段或无效值，Zod schema 在入口处拦截这些问题，返回结构化的错误信息帮助 LLM 自我修正，而不是让错误传播到搜索引擎内部引发难以理解的异常。

### 请求日志

HTTP 服务器对每个请求记录时间戳和工具描述信息。日志不仅用于调试，还为性能分析提供数据——通过对比请求到达时间和响应时间，运维人员可以识别慢查询和性能瓶颈。工具描述信息的记录则帮助理解 AI 助手的行为模式，例如它更倾向于使用 query 还是 get，搜索查询的复杂度分布如何。

### 本章小结

本章详细分析了 `src/mcp/server.ts` 这个 807 行的 MCP 服务器实现。两种传输协议——stdio 和 HTTP——分别面向桌面 AI 助手和网络集成场景。四个 MCP 工具覆盖了搜索（query）、单文档检索（get）、批量检索（multi_get）和状态查询（status）的完整能力谱。`qmd://{+path}` 资源模板提供了符合 MCP 规范的只读文档访问方式。`buildInstructions()` 通过动态生成系统提示，使 AI 助手能够根据索引的实际状态调整搜索策略。HTTP 服务器的会话模型在遵守 MCP 规范的同时，通过共享 store 实例最小化了资源消耗。Zod schema 验证为 LLM 生成的参数提供了可靠的运行时类型安全保障。
