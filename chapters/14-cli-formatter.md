## 第 14 章：CLI 与输出格式化

> 命令行界面是 qmd 与用户交互的第一触点。`src/cli/qmd.ts` 以 3187 行的篇幅实现了完整的 CLI 框架，而 `src/cli/formatter.ts` 用 430 行代码将搜索结果转化为六种输出格式。本章将深入这两个文件，分析 CLI 的架构选型、终端 UX 技巧以及格式化系统的设计。

### CLI 架构：无框架哲学

`src/cli/qmd.ts` 没有使用 Commander、Yargs 或 Oclif 等流行的 CLI 框架，而是直接采用 Node.js 内置的 `parseArgs()` 函数进行参数解析。这个选择意味着更少的依赖、更快的启动速度和更细粒度的控制权。

在启动阶段，第 104 行调用 `enableProductionMode()`，这个函数的核心作用是启用默认数据库路径——在生产模式下，如果用户没有显式指定 `dbPath`，qmd 会使用平台相关的默认位置（例如 `~/.qmd/index.sqlite`）。非生产模式（例如测试环境）则要求显式提供路径，避免意外操作用户的真实索引。

数据库和 store 的初始化采用了惰性模式。`getStore()` 和 `getDb()` 是两个延迟初始化函数，只在第一次调用时才真正打开数据库和创建 store 实例。这种设计使得不需要数据库的命令（如 `--help` 或 `--version`）可以瞬间返回，无需等待数据库连接建立。

YAML 配置文件与 SQLite 数据库的同步发生在首次访问时（第 117 至 125 行）。当 `getStore()` 被调用时，它检查是否存在 YAML 配置文件，如果存在则将配置内容同步到 SQLite 中，确保两个数据源的一致性。这个同步过程是单向的——YAML 作为源，SQLite 作为目标。

### 命令体系

qmd CLI 提供了十余个命令，按功能可以分为四个类别。

**搜索类命令**是使用频率最高的一组。`search` 执行纯 BM25 词法搜索，速度快、不需要 LLM，适合精确关键词匹配场景。`vsearch` 执行纯向量语义搜索，需要预先生成嵌入向量。`query` 是最强大的混合搜索命令，它先调用 LLM 进行查询扩展（生成同义词、相关术语、HyDE 假设文档），然后同时执行词法和向量搜索，最后对结果进行融合排序。

**检索类命令**用于直接获取文档。`get` 通过路径或文档 ID 获取单个文档的内容，`multi-get` 支持 glob 模式或列表式批量获取。

**管理类命令**操作集合和上下文。`collection add` 添加新集合，`collection remove` 删除集合，`collection rename` 重命名，`collection list` 列出所有集合。`context add` 和 `context remove` 管理集合的上下文描述。这些命令的底层都触发双写——同时更新 YAML 和 SQLite。

**运维类命令**负责索引维护。`embed` 为指定集合生成向量嵌入，`update` 重新扫描文件系统并更新索引，`status` 显示索引健康状态，`mcp` 启动 MCP 服务器，`maintenance` 执行数据库清理操作（如移除孤立记录、压缩数据库）。

### 终端 UX 系统

qmd CLI 在终端用户体验上投入了大量心思，这部分实现集中在 `qmd.ts` 的第 176 至 219 行。

**颜色系统**（第 176 至 186 行）遵循社区标准。首先检查 `NO_COLOR` 环境变量——如果设置了该变量，无论值是什么，都禁用所有颜色输出。这是 `no-color.org` 发起的跨工具约定。其次检查 `process.stdout.isTTY`——如果输出被管道到文件或另一个程序，也禁用颜色。只有当两个条件都不满足时，才启用 ANSI 颜色码。

**光标控制**（第 189 至 196 行）用于进度显示。在执行耗时操作（如向量嵌入生成）时，qmd 隐藏终端光标以避免闪烁，操作完成后恢复。这是一个小细节，但显著提升了视觉体验的专业感。

**OSC 9;4 进度条**（第 200 至 213 行）是一个高级终端特性。OSC（Operating System Command）9;4 是一个终端转义序列，允许程序在终端标题栏或标签页上显示进度百分比。支持此特性的终端（如 Windows Terminal、iTerm2）会在标签页上渲染一个小型进度条。qmd 在嵌入生成等长时间操作中发送 OSC 9;4 序列，即使用户切换到其他标签页，也能通过标签页上的进度指示了解任务进展。

**ETA 格式化**（第 216 至 219 行）由 `formatETA()` 函数处理。根据剩余时间的长短，它会选择合适的单位：秒级显示"23s"，分钟级显示"2m 15s"，小时级显示"1h 12m"。这种自适应格式避免了"3723 seconds remaining"这样不友好的显示。

### formatter.ts：六种输出格式

`src/cli/formatter.ts` 全文 430 行，定义了 `OutputFormat` 类型（第 36 行），支持六种输出格式。

**json 格式**输出美化的 JSON，每个搜索结果包含 `docid`（文档标识符）、`score`（相关性分数）、`file`（文件路径）、`title`（文档标题）、`context`（上下文片段）和 `snippet`（匹配代码片段）。这是程序化消费搜索结果的首选格式，也是与其他工具集成时最常用的数据交换格式。

**csv 格式**输出标准的逗号分隔值，首行为表头，后续行为数据。CSV 格式化需要处理三种特殊情况：字段包含逗号、字段包含双引号、字段包含换行符。`escapeCSV()` 函数（第 72 至 79 行）统一处理这些边界情况——如果字段包含逗号、双引号或换行符，就用双引号包裹整个字段，并将内部的双引号转义为两个双引号。

**xml 格式**输出 XML 文档，每个搜索结果作为一个 `<file>` 元素，属性包含元数据，子元素包含内容。`escapeXml()` 函数（第 81 至 88 行）处理标准的 XML 实体转义：`&` → `&amp;`、`<` → `&lt;`、`>` → `&gt;`、`"` → `&quot;`、`'` → `&apos;`。

**md 格式**输出 Markdown，使用标题标记文档、代码块展示内容。这种格式特别适合将搜索结果粘贴到文档或聊天工具中。

**files 格式**是最简洁的输出，每行一个结果，格式为 `docid,score,filepath`。适用于脚本消费，例如 `qmd search --format files "auth" | cut -d, -f3` 可以直接提取文件路径列表。

**cli 格式**是默认的终端输出格式，使用 ANSI 颜色码高亮显示分数、路径和匹配片段。这种格式在 `qmd.ts` 中单独处理，不经过 `formatter.ts` 的通用管道。

### 辅助函数

`formatter.ts` 还提供了几个通用的辅助函数。

`addLineNumbers()` 定义在第 56 至 59 行，为文本的每一行添加行号前缀，格式为 `N: `（行号加冒号加空格）。这个函数在显示代码片段时使用，帮助用户定位源文件中的具体位置。

`getDocid()` 定义在第 64 行，取文档内容哈希的前 6 个字符作为短标识符。6 位十六进制字符提供约 1600 万种可能，对于单个项目的文档集来说碰撞概率极低，同时保持了显示的简洁性。

### extractSnippet()：智能片段提取

`extractSnippet()` 的实现位于 `store.ts` 的第 3558 至 3624 行，虽然不在 CLI 或 formatter 模块中，但它是输出管道的关键组件，值得在此讨论。

该函数的核心算法是逐行评分。对文档的每一行，计算其与查询词和意图词的重叠度。查询词来自用户的搜索输入，意图词来自查询扩展阶段生成的语义标签。重叠度越高的行，得分越高。

选定最高分的行后，函数取该行的前后各 1 至 2 行作为上下文，形成一个小窗口。输出格式模仿了 `diff` 工具的样式，头部为 `@@ -start,count @@ (N before, M after)`，标明片段在原文档中的位置和上下文行数。

有一个重要的回退逻辑：如果按分块窗口提取的片段为空（例如查询词分散在文档各处、没有集中出现的区域），函数会回退到全文档搜索模式，在整个文档范围内重新寻找最佳匹配片段。这种降级策略确保了搜索结果始终附带有意义的代码片段，不会出现空白片段的尴尬情况。

### CLI 与 SDK 的关系

理解 CLI 和 SDK 的关系有助于把握 qmd 的整体架构。CLI（`qmd.ts`）是 SDK（`index.ts`）的上层消费者。CLI 的每个命令在内部都调用 `QMDStore` 接口的对应方法：`search` 命令调用 `store.searchLex()`，`query` 命令调用 `store.search()`，`update` 命令调用 `store.update()`。

CLI 的独特价值在于两方面。第一是用户交互：参数解析、颜色输出、进度显示、ETA 估算——这些都是终端环境特有的关注点，不属于 SDK 的职责范围。第二是输出格式化：将 SDK 返回的结构化数据转化为人类可读或机器可解析的文本，这是 `formatter.ts` 的核心使命。

### 本章小结

本章分析了 qmd 的命令行界面层。`src/cli/qmd.ts` 以 3187 行代码实现了完整的 CLI 框架，选择 Node.js 原生 `parseArgs()` 而非第三方框架，获得了更少的依赖和更快的启动速度。惰性初始化模式确保了不需要数据库的命令可以瞬间响应。终端 UX 系统在颜色管理、光标控制、OSC 9;4 进度条和 ETA 格式化等细节上展现了工程品质。`src/cli/formatter.ts` 以 430 行代码支持六种输出格式，从面向人类的彩色终端输出到面向程序的 JSON 和 CSV，覆盖了所有常见的消费场景。`extractSnippet()` 的逐行评分算法和 diff 风格输出为搜索结果提供了精准的代码定位能力。
